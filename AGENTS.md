# Agent Instructions

This repository is a vanilla Emacs configuration intended to live at
`~/.emacs.d`. It is public and reusable, so keep project instructions generic
and avoid personal machine names, account names, tokens, private paths, or
company-specific details.

## Project Shape

- `init.el` is deliberately small. It adds `lisp/` to `load-path`, requires the
  first-party modules, and loads `custom.el` last.
- `lisp/config-*.el` files are the source of truth for Emacs behavior. Add new
  durable behavior as a focused module or inside the closest existing module.
- `scripts/setup.el` owns package installation for a fresh clone.
- `scripts/compile.el` owns first-party byte-compilation.
- `scripts/host.sh` reports or installs host-level external tools.
- `scripts/user.sh` reports or applies user shell, editor, PATH, and tmux setup.
- `tests/init-test.el` is the ERT suite that protects the configuration
  contract.

## Editing Guidelines

- Keep files readable for humans. This config is meant to be read and adapted,
  not treated as opaque generated output.
- Prefer existing module patterns, `use-package`, built-in Emacs facilities, and
  small helper functions over broad rewrites.
- Add comments where they explain why a setting exists. Avoid comments that only
  restate the code.
- Keep generated state out of git. Do not commit `.elc` files, package
  directories, runtime caches, `custom.el`, or local personal agent settings.
- Do not edit files under `elpa/`, `quelpa/`, `var/`, `.cache/`, `eln-cache/`,
  or other runtime artifact directories.
- If a package is required by configuration, add it to
  `emacs-config-setup-packages` in `scripts/setup.el` and cover the behavior in
  tests.

## Verification

Run targeted checks after changing this repository:

```sh
make compile
make test
```

Use `make help` to see supported maintenance targets. Use `make host` and
`make user` as dry-run diagnostics; only use `HOST_INSTALL=1` or
`USER_INSTALL=1` when the user explicitly wants the helper to modify the host or
dotfiles.

## Common Workflows

- For Emacs Lisp changes, update the relevant `lisp/config-*.el` module and add
  or extend ERT coverage in `tests/init-test.el`.
- For agent-control changes, keep `config-agent.el`, `README.md`, and the ERT
  tests aligned. The primary user entry point is `C-c a ?`; context helpers
  should remain provider-neutral unless a provider truly needs special
  treatment.
- For new language/tool support, update `README.md`, `scripts/setup.el`,
  optional host-tool guidance in `scripts/host.sh`, and tests.
- For generated bytecode freshness, use `make compile`; do not commit the
  generated `.elc` files.
- For public-repo safety, keep credentials and personal preferences in local
  files that are ignored by git.
